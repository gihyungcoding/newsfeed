# 5. 마이크로서비스 분리 로드맵

현재는 단일 Spring Boot 애플리케이션(모듈러 모놀리스)이다. 바운디드 컨텍스트 간 통신을 포트와 이벤트로만 제한해 두었기 때문에, 아래 순서로 점진적으로 분리할 수 있다.

## 왜 처음부터 MSA로 하지 않았나

- 학습 초점이 뉴스피드 아키텍처(팬아웃, 캐시)에 있고, 서비스 간 네트워크 통신·배포·관측성은 별도의 큰 주제다.
- 잘못 그은 서비스 경계는 분산 모놀리스가 된다. 모놀리스 안에서 경계를 먼저 검증(ArchUnit로 강제)한 뒤 추출하는 것이 안전하다.

## 분리 단계

### Step 1 — Gradle 멀티모듈 전환

같은 배포 단위를 유지하면서 컨텍스트를 물리적 모듈로 분리한다.

```
newsfeed/
├── common/          # 공통 (인증, 예외)
├── user/            # user 컨텍스트
├── post/
├── engagement/      # 좋아요·답글·횟수
├── fanout/
├── feed/
└── app/             # 조립(assembly) 모듈 — 전체를 하나의 앱으로 부팅
```

- 컴파일 타임에 경계 위반이 원천 차단된다 (ArchUnit → Gradle 의존성 그래프로 승격).

### Step 2 — 팬아웃 워커 분리 (첫 번째 추출 대상)

- `fanout` 모듈 + Kafka 컨슈머만 담은 `fanout-worker` 앱을 별도 배포.
- 추출이 가장 쉬운 이유: 인바운드가 HTTP가 아니라 **Kafka뿐**이라 API 계약 변경이 없다. 팬아웃 부하(쓰기 폭증)를 API 서버와 독립적으로 스케일링할 수 있게 된다.
- 필요 작업: 팔로워 조회를 user 서비스 API 호출 또는 CQRS 읽기 모델(팔로우 이벤트 구독)로 전환.

### Step 3 — feed / post / engagement / user 서비스 분리

- `feed-service`(조회 전용, read-heavy → 독립 스케일링), `post-service`, `engagement-service`(좋아요·답글 쓰기 폭증 시 독립 스케일링), `user-service` 순으로 추출.
- API Gateway(Spring Cloud Gateway) 도입: 인증·처리율 제한을 게이트웨이로 이동 (책의 "웹 서버" 계층이 실제 게이트웨이가 됨).
- 서비스 간 동기 호출 최소화: 필요한 데이터는 이벤트 구독으로 각 서비스의 읽기 모델에 복제.

### Step 4 — 데이터 분리 및 운영 고도화

- 서비스별 DB 분리 (posts / users·follows / feed 읽기 모델).
- 피드 캐시 Redis Cluster 전환, 포스트 DB 샤딩(authorId 기준) 검토.
- 관측성: 분산 트레이싱(Micrometer + OTel), 서비스별 대시보드.
- 처리율 제한 고도화(토큰 버킷), 팬아웃 메시지 분할(팔로워 범위별 서브 태스크).

## Transactional Outbox 패턴 승격 기준

지금(4단계) `post-created` 이벤트는 애플리케이션 서비스가 DB 커밋 후 Kafka에 직접 발행한다. 이 방식은 "DB에는 저장됐는데 Kafka 발행이 실패"하는 틈이 있다 — 프로세스가 죽거나 브로커가 잠깐 응답하지 않으면 이벤트가 조용히 유실된다. 유실되면 그 포스트는 어떤 팔로워의 피드에도 팬아웃되지 않는다.

**지금은 감수한다**: 학습 프로젝트 규모에서 유실 확률은 낮고, 콜드 스타트 시 DB fallback(§3.4)이 있어 완전히 못 보게 되지는 않는다.

**실무에서 승격해야 하는 시점**:
- 이벤트 유실이 사용자에게 보이는 결함(피드 누락)으로 이어지고, 그 빈도가 SLA를 위협할 때
- 서비스 간 이벤트가 정산·알림 등 되돌리기 어려운 후속 작업을 트리거하기 시작할 때
- Step 3에서 post-service를 분리해 "발행 API"와 "이벤트 발행"이 진짜 다른 신뢰성 요구를 갖게 될 때

**전환 방법**: `posts` 테이블과 같은 트랜잭션에 `outbox` 테이블을 두고 이벤트를 함께 INSERT한다. 별도 릴레이(폴링 스케줄러 또는 Debezium CDC)가 outbox를 읽어 Kafka로 발행하고 발행 완료를 마킹한다. 커밋되면 반드시 발행되므로(at-least-once) 유실은 없어지지만, 팬아웃 워커는 이제 같은 이벤트를 중복 수신할 수 있다는 전제로 멱등성을 갖춰야 한다 — 우리 워커는 `ZADD`가 멱등이라(§3.3) 이 조건을 이미 만족한다.

## 분리 판단 기준

각 단계는 실제 필요(부하, 배포 주기 차이, 팀 분리)가 생겼을 때 진행한다. 이 프로젝트에서는 학습 목적상 Step 1~2까지를 후속 목표로 둔다.
