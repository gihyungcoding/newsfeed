# 1. 요구사항 정의 및 규모 추정

> 『가상 면접 사례로 배우는 대규모 시스템 설계 기초』 11장 "뉴스피드 시스템 설계"를 기반으로 한다.

## 1.1 기능 요구사항

| # | 요구사항 | 설명 |
|---|---------|------|
| F1 | 피드 발행 | 사용자는 새 포스트(스토리)를 발행할 수 있다 |
| F2 | 뉴스피드 조회 | 사용자는 팔로우한 사람들의 포스트를 모아 볼 수 있다 |
| F3 | 정렬 순서 | 피드는 **역시간순(reverse chronological)** 으로 정렬한다 (랭킹 알고리즘은 범위 제외) |
| F4 | 팔로우 관계 | 사용자는 다른 사용자를 팔로우/언팔로우할 수 있다 (최대 5,000명) |
| F5 | 콘텐츠 | 텍스트 포스트를 지원한다 (이미지/비디오는 설계에만 언급, 구현 제외) |
| F6 | 좋아요 | 사용자는 포스트에 좋아요를 누르거나 취소할 수 있다 |
| F7 | 답글 | 사용자는 포스트에 답글을 달 수 있다 |
| F8 | 횟수 표시 | 피드/포스트에 좋아요 횟수와 답글 횟수를 표시한다 |

## 1.2 비기능 요구사항

| # | 요구사항 | 설명 |
|---|---------|------|
| N1 | 낮은 조회 지연 | 피드 조회는 캐시 히트 시 수십 ms 내에 응답해야 한다 |
| N2 | 발행-노출 지연 허용 | 포스트 발행 후 팔로워 피드에 나타나기까지 약간의 지연(초 단위)은 허용한다 → **최종적 일관성** |
| N3 | 높은 읽기 비율 | 뉴스피드는 읽기가 쓰기보다 압도적으로 많다 (read-heavy) |
| N4 | 확장 가능한 구조 | 팬아웃/조회 경로를 수평 확장 가능한 구조로 설계한다 |

## 1.3 개략적 규모 추정 (책 기준)

- **DAU**: 1,000만 명
- **사용자당 친구(팔로우)**: 최대 5,000명
- **피드 조회**: 사용자당 하루 평균 5회 조회로 가정
  - 피드 조회 QPS ≈ `10,000,000 × 5 / 86,400 ≈ 약 580 QPS` (피크는 2~3배로 가정 → 약 1,500 QPS)
- **포스트 발행**: 사용자당 하루 평균 0.5개 발행으로 가정
  - 발행 QPS ≈ `10,000,000 × 0.5 / 86,400 ≈ 약 58 QPS`
- **읽기:쓰기 비율 ≈ 10:1 이상** → 읽기 경로 최적화(캐시, 팬아웃 쓰기)가 핵심

### 팬아웃 쓰기 부하 추정

포스트 1건 발행 시 평균 팔로워 100명에게 팬아웃한다고 가정하면:

```
팬아웃 쓰기 QPS ≈ 58 QPS × 100 = 약 5,800 writes/sec (Redis 기준 무리 없는 수준)
```

단, 팔로워 수백만 명의 유명인(celebrity)이 발행하면 단건에 수백만 건의 쓰기가 발생한다.
→ **핫키(hotkey) 문제**. 이것이 하이브리드 팬아웃 전략을 채택하는 이유다. ([03-detailed-design.md](03-detailed-design.md) 참고)

## 1.4 이번 구현의 범위

| 포함 | 제외 (설계 문서에만 언급) |
|------|--------------------------|
| 사용자 생성/조회, 팔로우/언팔로우 | 회원가입/로그인, OAuth 등 실제 인증 |
| 텍스트 포스트 발행 | 이미지/비디오 업로드, CDN |
| 좋아요/답글 + 횟수 캐시 (책의 5계층 캐시 전체 구현) | 알림(notification) 서비스 |
| 팬아웃 쓰기 (Kafka + 워커) | 피드 랭킹 알고리즘 |
| 하이브리드 팬아웃 (celebrity pull) | 다중 리전, 샤딩 |
| 뉴스피드 조회 (Redis 캐시 + 병합) | 그래프 DB (RDB로 대체) |
| 단순화한 인증(`X-USER-ID` 헤더)과 처리율 제한 | |

**단순화 결정의 근거**는 [03-detailed-design.md](03-detailed-design.md)의 트레이드오프 섹션에 기록한다.
